## AWS

### 1. Amazon EC2: Servidores en la Nube
**_(Elastic Compute Cloud)_** Es un serivicio de Amazon que nos proporciona servidores en la nube. **_Elastic_** hace referencia a la capicidad de aumentar los recursos (como CPU o RAM) de forma rápida y sencilla según las necesidades de nuestro proyecto.

#### ¿Para qué sirve EC2?
Nos permite ejeuctar aplicacion sin la necesidad de comprar o mantener software físico:
* Lanzar servidores en minutos con el sistema operativo de tu elección (Linux, Windows, etc.).
* Escalar recursos automáticamente para soportar picos de tráfico.
* Pagar solo por lo que usas, eliminando costos de infraestructura local.

#### ¿Cuándo se usa?
Lo utilizamos cuando necesitamos control total sobre el servidor, o las necesidades de proccesamiento de la aplicación varían con el tiempo. Es ideal cuando no queremos invertir en servidores físicos que se ponen obsoletos o infrautilizados.

#### Tipos de instancias Familias
AWS agrupa sus servidores en familias según el recurso que más destacan:
1. **_Propósito General (Familias T y M)_**
Son como un auto sedán estándar. Sirven para casi todo y son las más usadas.
    * **_¿Cómo funcionan?_** Tienen un poco de todo: buena CPU, buena RAM y buena red.
    * **_Diferencia clave:_**
        * La T (de Tiny o Turbo) es económica y "ahorra" energía para cuando necesites un acelerón de potencia.
        * La M (de Main) es más robusta y mantiene una potencia constante.
    * **_Uso ideal:_** Tu primera página web, un blog o un sistema interno de una oficina pequeña.
2. **_Optimizadas para Cómputo (Familia C)_**
Son como un auto de carreras. Todo se trata de la velocidad del motor (el procesador).
    * **_¿Cómo funcionan?_** Tienen procesadores mucho más potentes que las demás, pero no tanta memoria RAM. Están hechas para hacer cálculos matemáticos pesados muy rápido.
    * **_Uso ideal:_** Un servidor de videojuegos (donde la posición de los jugadores debe calcularse al instante) o para codificar videos.
3. **_Optimizadas para Memoria (Familias R y X)_**
Son como un autobús con muchísimos asientos. Aquí lo que importa es el espacio (la RAM).
    * **_¿Cómo funcionan?_** Tienen muchísima memoria RAM. Se usan cuando tu aplicación necesita "recordar" o tener a la mano gigas y gigas de datos sin tener que ir a buscarlos al disco duro (que es más lento).
    * **_Uso ideal:_** Bases de datos gigantes o aplicaciones que analizan datos en tiempo real (como la bolsa de valores). La R viene de RAM.
4. **_Optimizadas para Almacenamiento (Familias I y D)_**
Son como un camión de mudanza. Su fuerte es el "maletero" (el disco duro).
    * **_¿Cómo funcionan?_** Tienen discos duros especiales (SSD muy rápidos) que pueden leer y escribir archivos a una velocidad increíble.
    * **_Uso ideal:_** Si tienes que guardar y leer millones de archivos pequeñitos por segundo o si manejas archivos de video de tamaño masivo.
5. **_Computación Acelerada (Familias P y G)_**
Son como un transbordador espacial. Tienen componentes extra que los demás no tienen: las GPUs (tarjetas gráficas).
    * **_¿Cómo funcionan?_** No solo usan el procesador normal, sino que usan el poder de las tarjetas gráficas para tareas que un procesador normal tardaría años en hacer.
    * **Uso ideal:** Inteligencia Artificial (entrenar a un ChatGPT), diseño 3D profesional o minería de datos compleja. La G viene de Graphics.

### 2. Amazon RDS: Bases de Datos Gestionadas
(Relational Database Serivice) Es el hermano inteligente de C2 pero especializado para Bases de Datos Relacionales.
RDS es un servicio de bases de datos gestionadas donde AWS se encarga de la infraestructura. No administras el
servidor, solo la base de datos. Ideal para aplicaciones que necesitan persistencia confiable.

#### Diferencia entre en EC2 y Amazon RDS
| Característica | Base de Datos en EC2 (Tú la gestionas) | Amazon RDS (Gestionada por AWS) |
|---|---|---|
| Instalación | Tienes que instalar el motor (MySQL, etc.) a mano. | Viene lista y configurada en pocos minutos. |
| Mantenimiento | Tú instalas parches y actualizaciones del sistema. | AWS se encarga de las actualizaciones y parches. |
| Copias de Seguridad | Tienes que programarlas y gestionarlas tú. | Automáticas. Puedes "viajar en el tiempo" a un punto exacto. |
| Alta Disponibilidad | Muy difícil de configurar manualmente. | Se activa con un clic (Multi-AZ). |
| Control Total | Tienes acceso al Sistema Operativo (SSH). | No tienes acceso al SO, solo a la base de datos. |
| Escalabilidad | Tienes que mover los datos a un nuevo servidor. | Cambias el tamaño de la instancia con un par de clics. |
| Responsabilidad | Todo es tu responsabilidad (seguridad, backups, OS). | Tú solo te encargas de tus datos y el diseño de tablas. |

#### ¿Por qué elegir una u otra?
* Elige EC2 solo si necesitas una versión muy específica de una base de datos que RDS no soporte, o si necesitas acceso total al sistema operativo para instalar herramientas extra.
* Elige RDS para el 95% de los casos. Te ahorra semanas de trabajo técnico y reduce el riesgo de que pierdas tus datos por un error humano.

#### El uso más común
Tener tu aplicación corriendo en una instancia EC2 (el servidor web) y conectarla a una instancia RDS (donde viven los datos). Al estar separados, si tu servidor web falla, tus datos siguen intactos y seguros en RDS.

### 3. Amazon VPC: Tu Red Privada
Con las VPC construimos nuestra propia "urbanización privada" con muros, guardias y reglas de quién entra y quién sale.

#### 1. ¿Qué es y Para qué sirve?
Es una red virtual dedidcada a tu cuenta de AWS. Sirve para que tengamos un control total sobre nuestro entorno de red, Nosotros decidimos:
* Qué rango de direcciones IP usar.
* Quién puede entrar desde internet y quién no.
* Cómo se comunican tus servidores entre sí de forma privada (sin salir a internet). 

#### 2. Conceptos clave (Los componentes de tu red)
Siempre que utilicemos una VPC debemos pensar en estos cuatro conceptos:
1. **_Subredes (Subnets):_** Son las divisiones dentro de nuestra red:
    * **_Pública_:** Tiene acceso directo a internet (aquí pones tu servidor web).
    * **_Private_:** No tiene acceso a internet (aquí escondemos nuestra base de datos RDS y los workes (Esclavos) para que nadie la ataque).
1. **_Internet Gateway (IGW):_** Es la puerta a la calle. Si no conectamos una, nuestra VPC estara totalmente aislada del mundo.
1. **_Tabla de ruteo (Route Table):_** Es el GPS de la red. Le dice a los datos: "si quieres ir a internet, ve por aquella puerta; si quieres ir al servidor de al lado, ve por este pasillo".
1. **_Security Groups (Grupos de Seguridad):_** Son nuestros porteros virtuales, es un muro que rodea a cada servidor y dice: "Solo dejo pasar a personas que vengan por el puerto 80 (web)".

#### 3. Comparativa: Red Tradicional vs. Amazon VPC
| Característica | Red Física (On-Premise / Oficina) | Amazon VPC (Nube de AWS) |
|---|---|---|
| Costo Inicial | Muy alto (Inversión en routers, switches y cables). | $0 (Crear la VPC es gratis; solo pagas extras como NAT Gateways). |
| Tiempo de Despliegue | Días o semanas (Instalación física y cableado). | Segundos (Se configura mediante software o clics). |
| Escalabilidad | Limitada al hardware físico disponible. | Casi infinita y bajo demanda. |
| Seguridad | Depende de firewalls físicos y acceso al rack. | Definida por software (Security Groups y NACLs). |
| Mantenimiento | Requiere personal técnico para arreglar cables/hardware. | Cero mantenimiento de hardware (AWS se encarga). |
| Flexibilidad | Cambiar la arquitectura requiere reconectar cables. | Cambias subredes y rutas con un par de clics. |

#### 4. Tip para tu aprendizaje:
Si estás guardando estas tablas, recuerda que la VPC es la base de todo en AWS. Sin una red (VPC) bien configurada, no puedes conectar tus servidores (EC2) con tus bases de datos (RDS) de forma segura.

### 4. AWS Lambda:Computación Serverless
Si EC2 es rentar un coche (tú lo conduces, le pones gasolina y lo estacionas) y RDS es tener un chef, AWS Lambda es como pedir un Uber: tú solo dices a dónde quieres ir, el coche aparece, te lleva y desaparece. No te importa el motor, el seguro ni quién lo conduce.

#### 1. ¿Qué es? (El concepto "Serverless")
Es un servicio de computación que nos permite ejecutar código sin gestionar servidores. 
"Serverless" no significa que no haya servidores (claro que los hay, son de Amazon), significa que tú no los ves ni los administras.
Nosotros solo subimos nuestro código (Python, Node.js, Java, etc.) y Lambda se encarga de ejecutarlo cuando sea necesario. 

#### 2. ¿Para qué sirve?
Sirve para ejecutar tareas específicas en respuesta a eventos. 
Ejemplo: Alguien sube una foto a tu web, Lambda se activa automáticamente, Cambia el tamaño de la foto, Lambda se apaga.

#### 3. ¿Cuándo se usa?
Se usa cuando tenemos tareas cortas y rápidas. No es para tener un servidor encendido 24/7, sino para "reaccionar" a cosas que pasan. 
* **_Procesamiento de datos:_** Validar un formulario.
* **_Backends de apps:_** Cuando alguien pulsa un botón en una app móvil.
* **_Automatización:_** Apagar tus instancias EC2 por la noche para ahorrar dinero.

#### 4. ¿Qué tipos hay? (Configuraciones)
En Lambda no elegimos "familias" como en EC2, solo elegimos:
1. Memoria RAM: Desde 128 MB hasta 10 GB. (AWS asigna potencia de CPU proporcionalmente a la RAM que elijas).
1. Tiempo de ejecución (Timeout): El máximo que puede durar una tarea son 15 minutos. Si tu código tarda más, Lambda no es para ti. 

#### 5. Usos más comunes
1. Webhooks: Recibir notificaciones de otros servicios (como un pago de PayPal).
1. Limpieza de archivos: Borrar archivos viejos de una carpeta cada domingo.
1. Chatbots: Procesar lo que el usuario escribe y responder.
1. Internet de las Cosas (IoT): Recibir la temperatura de un sensor y guardarla en una base de datos.

La Regla de Oro para principiantes:
¿Necesitas un programa corriendo todo el día (como un sitio web tradicional)? Usa EC2.
¿Necesitas que algo pase solo cuando ocurre un evento (como enviar un email al registrarse un usuario)? Usa Lambda.
