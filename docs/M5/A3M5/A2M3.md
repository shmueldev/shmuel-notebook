## Consola de AWS y el SDK (Boto3)
La diferencia es como interactúamos con los servicios, **_La consola_** es una interfaz visual (clics), mientras que el **_SDK_** es una herramienta de programación (código).

### ¿Qué es cada uno?
* **Consonla de administración de AWS** Es una aplicación gráfica que permite gestionar recursos de AWS (como servidores EC2 o bases de datos RDS) de forma visual e intuitiva desde el navegador.
* **SDK (Software Development Kit)-Boto3** Es una bibloteca especifica para python, Nos permite crear código para crear, configurar y gestionar servicios de AWS automáticamente sin usar la web.

### ¿Dónde y cuándo se utilizan?
1. **Consola de AWS:**
    * Aprendizaje: Para explorar nuevos servicios por primera vez.
    * Configuración puntual: Tareas rápidas que no se repetirán (ej. cambiar una contraseña).
    * Monitoreo: Ver gráficos de rendimiento o revisar facturación.
1. **SDK Boto3 (Python):**
    * Aplicaciones: Cuando tu app necesita subir fotos a un "bucket" de S3 o leer datos de una base de datos DynamoDB automáticamente.
    * Automatización: Scripts que apagan servidores los fines de semana para ahorrar costos.
    * DevOps: Integración en tuberías de despliegue continuo (CI/CD). 

**Resumen:** Usa la Consola para "ver y tocar" lo que estás haciendo de forma manual; usa Boto3 cuando necesites que tu código hable directamente con AWS para automatizar procesos.

### Ejemplo de la vida real
Para este ejemplo, compararemos una tarea clásica: Crear un "Bucket" en Amazon S3 (un espacio para guardar archivos) y subir un archivo de texto.
1. **_Proceso en la Consola de AWS (Manual)_** Para hacer esto visualmente, tendrías que:
    + Iniciar sesión en el navegador.
    + Buscar el servicio S3.
    + Hacer clic en el botón naranja "Create bucket".
    + Escribir un nombre único (ej. mi-bucket-prueba-123).
    + Elegir la región y hacer clic en "Create" al final de la página.
    + Entrar al bucket, darle a "Upload", buscar tu archivo en la PC y confirmar.

2. **_Proceso con SDK Boto3 (Python)_**
Con Boto3, traducimos todos esos clics en unas pocas líneas de código. Una vez que lo tenemos, puedes correrlo mil veces o para mil buckets distintos sin tocar el mouse.

```py
import boto3
from botocore.exceptions import ClientError

# 1. Crear el cliente de S3
s3 = boto3.client('s3', region_name='us-east-1')

bucket_name = 'mi-bucket-prueba-boto3-2024' # Debe ser único globalmente
archivo_local = 'notas.txt'
nombre_en_s3 = 'archivo_en_la_nube.txt'

try:
    # 2. Crear el Bucket
    print(f"Creando bucket: {bucket_name}...")
    s3.create_bucket(Bucket=bucket_name)
    
    # 3. Subir un archivo
    print(f"Subiendo {archivo_local}...")
    s3.upload_file(archivo_local, bucket_name, nombre_en_s3)
    
    print("¡Éxito! El proceso terminó correctamente.")

except ClientError as e:
    print(f"Error detectado: {e}")
```

### ¿Qué pasó aquí?
1. boto3.client('s3'): Es como abrir la pestaña de S3 en tu navegador.
1. create_bucket(): Es el equivalente al botón naranja "Create bucket".
1. upload_file(): Es el equivalente a arrastrar el archivo y darle a "Upload".

### Ventaja real de Boto3:
Imagina que tienes que subir 500 imágenes que están en diferentes carpetas. En la consola tardarías horas haciendo clics. En Boto3, solo metes la función upload_file dentro de un ciclo for y el script lo hace en segundos mientras te tomas un café.

## Repaso de Triggers - Proccedures - 
Vamos a repasar estos en SQL

### Trigers
Es un disparador, es un objeto de la base de datos que funciona como **_vigilante_**. Es un bloque de código que se ejecuta automáticamente cuando ocurre un evento (insertar, actualizar, borrar) en una tabla.

### ¿Cúando se usan los trigers?
Se utilizan para mantener la integridad de los datos y automatizar tareas que no se deben olvidarse:
* **Auditoría:** Guardar quine borró un registro y a qué hora en una tablaF de Log
* **Validaciones Complejas:** Impedir que el cliente inserte un pedido si tiene deudas (Reglas de negocio)
* **Sincronización:** Si actualizamos el precio de un producto, el trigger puede actualizar automáticamente el total de las facturas pendientes
* **Derivación de datos** Calcular automáticamente el IVA cuando se inserta un precio neto.

### Tipos de trigers
Se clasifican según cúando actúan y qué acción los activa:
**Por el momento de la ejecución**
1. **BEFORE** Se ejecutan antes de que el cambio e guarde en la tabla, ideal para validar datosy modificarlos antes de que entren.
1. **AFTER** Se ejecuta después que la operación se completo con éxito. Ideal paa auditorias o actualizar otras tablas.
1. **INSTEAD OF (En lugar de)**: Se usa mayormente en Vistas. Salta el comando original y hace lo que dice el trigger. 
**Por el evento que lo dispara:**
1. **INSERT:** Al crear nuevos registros.
1. **UPDATE:** Al modificar registros existentes.
1. **DELETE:** Al eliminar registros.

### Estructura y Ejemplo (Sintaxis Estándar)
Imagina que quieres que, cada vez que alguien borre un producto, se guarde un aviso en una tabla de historial_borrados.
```sql
CREATE TRIGGER log_borrado_productos
AFTER DELETE ON productos -- Se dispara DESPUÉS de borrar en la tabla productos
FOR EACH ROW              -- Se ejecuta por cada fila borrada
BEGIN
    INSERT INTO historial_borrados (nombre_producto, fecha_borrado)
    VALUES (OLD.nombre, NOW()); -- 'OLD' representa el dato que se está borrando
END;
```
### Conceptos clave dentro del código:
1. **NEW:** Es una "tabla virtual" que contiene los datos nuevos que están entrando (se usa en INSERT y UPDATE).
1. **OLD:** Es una "tabla virtual" que contiene los datos viejos que ya estaban o se están borrando (se usa en UPDATE y DELETE). 

### Stored Procedure (Procedimiento Almacenado)
Es un conjunto de instrucciones en SQL que guardamos en la bse de datos, para reutilizar en cualquier momento. A diferencia del triger que es automático, el procedimiento es manual lo creamos, luego lo llamamos, lo ejecutamos cada vez que lo necesitemos.

### ¿Qué es y cuándo se usa?
Piénsalo como una función o un script que vive dentro de tu base de datos.
**Encapsulamiento:** En lugar de escribir una consulta de 50 líneas cada vez, guardas el código en un procedure y solo usas una línea para ejecutarlo.
**Seguridad:** Puedes dar permiso a alguien para que use un procedure pero no para que vea las tablas directamente.
**Rendimiento:** El código ya está "pre-compilado" en el servidor, por lo que corre un poco más rápido que enviar mil consultas sueltas.

### Tipos de parametros
Esta es la parte más importante, porque podemos enviarle datos al procedure para que trabaje con ellos.
1. **IN (Entrada)** Datos que le pasamos al procedimiento (ej. el ID de un usuario)
1. **OUT (Salida)** El procedimiento nos devuelve un dato (ej. el total de una venta)
1. **INOUT (Mixto)** Un valor que entra, se modifica y sale actualizado.

### Ejemplo Práctico (Sintaxis)
Queremos un procedimiento que al darle el **ID de un producto** y un **Porcentaje**, le suba el precio automaticamente
```sql
CREATE PROCEDURE AumentarPrecio(
    IN producto_id INT,      -- Parámetro de entrada: ¿A cuál producto?
    IN porcentaje DECIMAL    -- Parámetro de entrada: ¿Cuánto le subimos?
)
BEGIN
    UPDATE productos 
    SET precio = precio + (precio * porcentaje / 100)
    WHERE id = producto_id;
END;
```
¿como llamamos a los procedimientos?
```sql
CALL AumentarPrecio(10, 15); -- Súbele el 15% al producto con ID 10
```
**Resumen en Tabla Markdown**
| Aspecto | Detalle |
|---|---|
| Invocación | Explícita por el usuario (CALL). |
| Parámetros | Flexibles (puedes pasarle variables). |
| Retorno | Puede devolver valores o conjuntos de resultados (tablas). |
| Mantenimiento | Centraliza la lógica: si cambias el procedure, cambia en todas las apps. |

### Vamos a crear un escenario completo: un sistema de Ventas.
   1. Usaremos un Stored Procedure para registrar una venta (acción manual).
   2. Usaremos un Trigger para que, cada vez que se registre esa venta, se reste automáticamente el stock del producto (acción automática).

1. Preparación: Las Tablas
Primero necesitamos dos tablas básicas:
```sql
-- Tabla de productos con su inventarioCREATE TABLE productos (
    id INT PRIMARY KEY,
    nombre VARCHAR(50),
    stock INT

-- Tabla para registrar las ventas realizadasCREATE TABLE ventas (
    id INT AUTO_INCREMENT PRIMARY KEY,
    producto_id INT,
    cantidad INT,
    fecha DATETIME
-- Insertamos un producto de pruebaINSERT INTO productos (id, nombre, stock) VALUES (1, 'Laptop Gaming', 10);
```
2. El Trigger (El Automatismo)
Este trigger "vigila" la tabla ventas. En cuanto ve que entra una fila nueva (INSERT), va a la tabla productos y descuenta el stock.
```sql
CREATE TRIGGER actualizar_stock_post_venta
AFTER INSERT ON ventas -- Se dispara DESPUÉS de que se inserta la ventaFOR EACH ROWBEGIN
    -- Usamos NEW.producto_id para saber qué se vendió
    UPDATE productos 
    SET stock = stock - NEW.cantidad 
    WHERE id = NEW.producto_id;END;
```

3. El Stored Procedure (La Lógica Manual)
Este procedimiento recibe el ID del producto y cuántos se vendieron. Valida que haya stock y, si todo está bien, inserta la venta.
```sql
CREATE PROCEDURE RegistrarVenta(
    IN p_id INT, 
    IN p_cantidad INT
)BEGIN
    -- Insertamos en la tabla ventas
    -- Al hacer este INSERT, se disparará automáticamente el TRIGGER de arriba
    INSERT INTO ventas (producto_id, cantidad, fecha) 
    VALUES (p_id, p_cantidad, NOW());
    
    SELECT 'Venta registrada con éxito y stock actualizado' AS Mensaje;END;
```
------------------------------
4. ¿Cómo funciona todo junto?
Ahora, en lugar de hacer INSERT y UPDATE por separado, solo tienes que ejecutar una línea:
`-- Queremos vender 3 Laptops (ID 1)CALL RegistrarVenta(1, 3);` 

**_¿Qué pasó "detrás de cámaras"?_**

   1. Tú llamaste al Procedure: Le pasaste el ID 1 y la cantidad 3.
   2. El Procedure ejecutó el INSERT: Creó un registro en la tabla ventas.
   3. El Trigger se "despertó": Como detectó un INSERT en la tabla ventas, se ejecutó solo.
   4. El Trigger actualizó el stock: Vio que el NEW.cantidad era 3 y lo restó de la tabla productos.

Resultado final: Si consultas la tabla productos, verás que la 'Laptop Gaming' ahora tiene 7 unidades en lugar de 10.
